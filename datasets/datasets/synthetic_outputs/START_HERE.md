## What this is

These are sensor readings from five environmental monitoring sites in the
Eastern Cape, South Africa. The data covers two years (2022–2023) at
15-minute intervals — temperature, rainfall, soil moisture, water quality,
pest pressure, animal location, and security events.

The data is **synthetic** — generated by a calibrated model, not collected
from real sensors. It is built from real weather data for the location and
calibrated against real environmental standards (SANS 241, FAO, WHO). It
behaves the way real sensor data would: it has gaps, spikes, seasonal
patterns, and alerts that fire at the right times for the right reasons.

All datasets share the same temporal spine — a rain event affects water
quality, soil moisture, and livestock movement simultaneously. You can work
with a single dataset or explore relationships across domains.

Python code is provided for loading and validating the data, but there is
no requirement to use Python in your project. Use whatever tools and
languages are appropriate for what you are building.

## The files

Pick **one** dataset for your dashboard focus. You will be assigned or choose
this during the COIL workshops.

| File | What it contains |
|------|-----------------|
| `water_quality.csv` | pH, turbidity, conductivity, water temperature and level — 3 river/reservoir sites |
| `soil_monitoring.csv` | Soil moisture, pH, salinity, air temperature and humidity — 3 agricultural sites |
| `pest_monitoring.csv` | Temperature, humidity, leaf wetness, pest trap counts — 3 crop sites |
| `livestock_tracking.csv` | GPS location and accelerometer activity — 2 animal herds |
| `infrastructure_events.csv` | Gate and fence events — 1 farm gate node |
| `security_events.csv` | Motion, door, smoke, sound, flame, panic — 3 community nodes |

Each file has ~70,000 rows per site. All files share the same timestamps —
rain events affect water quality, soil moisture, and livestock movement
simultaneously across datasets.

## The two columns your dashboard must use

Every file has these two columns:

**`status`** — `"normal"` / `"warning"` / `"critical"`

This is the overall alert level for each reading. Your dashboard should
display this clearly for every site. It is derived from the sensor readings
using the thresholds in `alert_rules.md` — you do not need to compute it
yourself, but you do need to understand the rules in order to explain alerts
to users.

**`alert_triggered`** — `0` or `1`

`1` when status is `warning` or `critical`. Use this for filtering,
counting, and driving alert logic in your dashboard.

For the full threshold rules — what values trigger warning vs critical for
each sensor — see **`alert_rules.md`**. That document is your specification.
Implement those rules as your baseline. You do not need to change them.

## Missing data

The data contains gaps — roughly 2–5% of rows have `NaN` values in one or
more sensor columns. This is realistic: sensors go offline, LoRa packets
get lost, batteries run flat.

Your dashboard must handle this gracefully. Do not assume the data is
complete. Decide explicitly how to treat a gap — skip it, interpolate,
show "no data" — and be consistent.

```python
# Quick check: which columns have missing values, and how many?
print(df.isnull().sum()[df.isnull().sum() > 0])
```
